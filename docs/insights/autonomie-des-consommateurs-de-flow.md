# Autonomie des consommateurs de FLOW

## Constat

Les métiers ne demandent pas à FLOW de construire toutes les expériences utilisateurs du groupe.

Ils souhaitent conserver leurs outils d'engagement et leur autonomie de réalisation :

- Salesforce ;
- Salesforce Commerce Cloud ;
- Elastic ;
- portails B2B ;
- applications mobiles ;
- batchs métier ;
- futurs produits issus d'acquisitions.

## Insight

FLOW doit fournir des capacités métier réutilisables sans imposer les expériences qui les consomment.

FLOW fournit les capacités.

Les consommateurs construisent les expériences.

## Conséquences

Les capacités FLOW doivent être accessibles sous plusieurs formes :

- API unitaires ;
- API de masse ;
- événements ;
- exports et imports ;
- projections de données ;
- flux analytiques.

## Impact architectural

Les domaines consommateurs doivent rester autonomes dans le choix de :

- leurs interfaces ;
- leurs parcours ;
- leurs technologies ;
- leurs automatisations.

FLOW devient une plateforme de capacités partagées plutôt qu'une plateforme d'engagement.

## À retenir

L'autonomie des consommateurs constitue un objectif architectural explicite du programme FLOW.
